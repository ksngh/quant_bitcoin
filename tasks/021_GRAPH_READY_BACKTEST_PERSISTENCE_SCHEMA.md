# Task 021: Graph-Ready Backtest Persistence Schema

# Goal

Design the PostgreSQL schema changes needed to persist backtest runs in a graph-ready shape before adding write behavior to `quant-bitcoin-postgres-backtest`.

# Source Requirement

User request on 2026-05-16:

- Add the ability for `quant-bitcoin-postgres-backtest` to save backtest results to the database.
- Consider the design carefully because a later workflow will read this database and draw graphs.
- Follow `AGENTS.md` and define the immediate next tasks before implementation.

# Extracted Roles

- Owner role: Architect
- Supporting roles: Backtest Engine, Persistence, Strategy metadata, Test Designer, Documentation
- Forbidden roles: Live Trading, Real Binance order execution, Exchange account APIs, Dashboard, FastAPI, Streamlit, Scheduler, Risk Management, Portfolio Optimization, Machine Learning

# Context

Task 013 proposed persistence candidates for `strategy_configs`, `backtest_runs`, `backtest_trades`, and `backtest_results`. Task 020 added `quant-bitcoin-postgres-backtest`, which currently reads PostgreSQL candles, runs `RsiStrategy` through `BasicBacktester`, and prints deterministic JSON to stdout without writing results back to PostgreSQL.

Before write behavior is added, the project needs an accepted graph-ready persistence contract. The later graphing workflow should be able to read run metadata, summary metrics, simulated trades, and time-series points without re-running the backtest.

# Scope

- Review Task 013's proposed `strategy_configs`, `backtest_runs`, `backtest_trades`, and `backtest_results` design.
- Decide whether those tables are sufficient for graphing or whether a first-class time-series table is required.
- Design a graph-ready PostgreSQL schema for persisted backtest outputs.
- Include a deterministic per-run identity and relationships between runs, summaries, trades, and graph points.
- Include enough fields for later equity/price/position graphing without adding a dashboard.
- Define indexes and uniqueness rules needed for common reads such as latest runs, runs by symbol/interval/time range, trades by run, and graph points by run/time.
- Define JSON metadata boundaries for flexible but non-critical values.
- Define retention of stdout JSON behavior for the existing CLI unless a later implementation task explicitly changes it.
- Update `STATUS.md` during documentation if phase, step, goal, active task, blocker, open question, or completion state changes.

# Out of Scope

- Implementing database schema changes.
- Writing migrations or SQL files.
- Implementing repository classes.

- Changing RSI strategy behavior.
