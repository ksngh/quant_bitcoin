# Accepted Graph-Ready Persistence Design

This task accepts a first-class graph-ready backtest output schema for the next implementation task. Task 013's `strategy_configs`, `backtest_runs`, `backtest_trades`, and `backtest_results` tables remain directionally correct, but they are not sufficient for graphing by themselves because later graphing would have to re-read candles and replay state transitions. This task therefore adds a required `backtest_graph_points` table that stores one deterministic time-series point per candle included in a run.

## Design Decision

Graph points must be materialized when the backtest result is saved, not reconstructed by a later graph read workflow.

Rationale:

- Later graphing must read persisted output without re-running the strategy or backtest.
- Persisted trades alone are sparse and do not contain HOLD candles, so they cannot directly render equity, cash, position, and close-price lines for every candle.
- Reconstructing graph points on every graph read would couple graphing code to current backtest state-transition rules and to candle availability.
- Materializing graph points during result persistence keeps strategy code unaware of PostgreSQL and keeps market-data code responsible only for candles.
- `BasicBacktester.run(candles, strategy)` does not need to change for Task 022. The persistence layer can derive graph points after the run from the in-memory standard candles and the returned `BacktestResult.trades` by walking candles in timestamp order and applying each simulated trade at its candle timestamp.

If a future backtester supports multiple trades at the same timestamp or intra-candle execution, that future task must extend the graph-point materialization contract explicitly. The current accepted design is sufficient for the existing long-only fixed-quantity RSI backtester.

## Boundary Rules

- Candle input identity remains `candles.source + candles.symbol + candles.interval + candles.open_time` and is owned by Market Data.
- Backtest output identity is a separate deterministic `run_key` on `backtest_runs` and is owned by Backtest/Persistence.
- Strategy code may expose deterministic configuration values, but it must not read from or write to PostgreSQL.
- Market-data code must not generate signals, simulated quantities, summaries, trades, or graph points.
- The existing CLI stdout JSON behavior must remain unchanged unless a later assigned task explicitly changes it. Task 022 may add opt-in database-save behavior while still printing the current deterministic JSON payload.

## Deterministic Run Identity

`backtest_runs.run_key` is the stable per-run identity for idempotent persistence and duplicate-safe saves. It should be a deterministic hash, for example SHA-256 hex, over a canonical JSON object containing at least:

- run schema version, for example `backtest_persistence_schema_v1`,
- backtest engine name and version, for example `BasicBacktester` and an implementation/config version,
- strategy name and version,
- canonical strategy parameters,
- candle source, symbol, and interval,
- requested start and end filters,
- actual first and last candle open times included,
- candle count,
- starting cash,
- trade quantity.

The hash must not include volatile values such as `created_at`, database surrogate IDs, stdout formatting, or local environment paths. If the same deterministic run is saved again, Task 022 should either no-op or replace the same run atomically according to the implementation task's accepted behavior.

## `strategy_configs`

Owner role: Strategy metadata.

Purpose: Store deterministic strategy parameters referenced by backtest runs.

Fields:

| Field | Meaning |
| --- | --- |
| `id` | Surrogate primary key. |
| `strategy_key` | Stable strategy identifier such as `rsi`. |
| `strategy_name` | Human-readable strategy class/name such as `RsiStrategy`. |
| `version` | Strategy/config contract version, initially a simple project-controlled string. |
| `parameters` | Canonical JSONB strategy parameters, initially `window`, `buy_threshold`, and `sell_threshold`. |
| `parameters_hash` | Deterministic hash of canonical `parameters` for uniqueness and lookup. |
| `metadata` | Optional non-critical JSONB notes. |
| `created_at` | Row creation timestamp. |

Primary key:

- `id`.

Uniqueness:

- Unique `strategy_key + version + parameters_hash`.

Indexes:

- Unique index on `strategy_key, version, parameters_hash`.
- Optional lookup index on `strategy_name` if human-facing filtering is later needed.

JSON boundary:

- `parameters` contains values that affect deterministic run identity.
- `metadata` must not contain values required to reproduce or identify a run.

## `backtest_runs`

Owner role: Backtest/Persistence.

Purpose: Store one deterministic backtest execution and connect input candles, strategy configuration, summary, trades, and graph points.

Fields:

| Field | Meaning |
| --- | --- |
| `id` | Surrogate primary key. |
| `run_key` | Deterministic hash identity for this exact run. |
| `strategy_config_id` | Foreign key to `strategy_configs.id`. |
| `engine_name` | Backtest engine identifier, initially `BasicBacktester`. |
| `engine_version` | Backtest engine contract/version string. |
| `candle_source` | Stored candle source, for example `binance_spot`. |
| `symbol` | Stored candle symbol, for example `BTCUSDT`. |
| `interval` | Stored candle interval, for example `1m`. |
| `requested_start_time` | Optional requested first candle open time filter. |
| `requested_end_time` | Optional requested last candle open time filter. |
| `actual_start_time` | First candle open time actually included, nullable for empty runs. |
| `actual_end_time` | Last candle open time actually included, nullable for empty runs. |
| `candle_count` | Number of candles included. |
| `starting_cash` | Starting simulated cash. |
| `trade_quantity` | Fixed simulated trade quantity for the current backtester. |
| `status` | Run status such as `completed` or `failed`; graph-ready reads should use completed runs only. |
| `metadata` | Optional non-critical JSONB run metadata. |
| `created_at` | Row creation timestamp. |
| `completed_at` | Completion timestamp for completed runs. |

Primary key:

- `id`.

Foreign keys:

- `strategy_config_id` references `strategy_configs(id)`.

Uniqueness:

- Unique `run_key`.

Indexes:

- Unique index on `run_key` for idempotent saves and direct run lookup.
- Descending index on `created_at` for latest runs.
- Composite index on `candle_source, symbol, interval, created_at DESC` for latest runs by market.
- Composite index on `candle_source, symbol, interval, actual_start_time, actual_end_time` for runs covering a candle time range.
- Index on `strategy_config_id, created_at DESC` for strategy-specific run history.

JSON boundary:

- `metadata` may contain CLI invocation labels, notes, code revision, or stdout payload references if useful.
- Required query fields and deterministic identity inputs must remain first-class columns or `strategy_configs.parameters`, not only metadata.

## `backtest_results`

Owner role: Backtest/Persistence.

Purpose: Store the one-row summary for a completed run.

Fields:

| Field | Meaning |
| --- | --- |
| `backtest_run_id` | Primary key and foreign key to `backtest_runs.id`. |
| `starting_cash` | Starting simulated cash from the result summary. |
| `ending_cash` | Ending simulated cash. |
| `ending_position` | Ending simulated position. |
| `final_price` | Last candle close price, nullable for empty runs. |
| `final_equity` | Ending equity. |
| `total_return` | Deterministic total return. |
| `trade_count` | Number of simulated trades. |
| `buy_count` | Number of simulated BUY trades. |
| `sell_count` | Number of simulated SELL trades. |
| `metadata` | Optional non-critical JSONB result metadata. |
| `created_at` | Row creation timestamp. |

Primary key:

- `backtest_run_id`.

Foreign keys:

- `backtest_run_id` references `backtest_runs(id)` with delete behavior chosen by the implementation task. If deletes are allowed, dependent results, trades, and graph points should be deleted together.

Uniqueness:

- One result row per run by primary key.

Indexes:

- The primary key is sufficient for the current read path because results are read by run.
- Optional indexes on `final_equity` or `total_return` are out of scope until ranking/filtering requirements exist.

## `backtest_trades`

Owner role: Backtest/Persistence.

Purpose: Store sparse simulated trade markers for a run.

Fields:

| Field | Meaning |
| --- | --- |
| `id` | Surrogate primary key. |
| `backtest_run_id` | Owning run. |
| `sequence` | Deterministic zero- or one-based trade order within the run. |
| `candle_open_time` | Candle open timestamp that generated the trade. |
| `signal` | Simulated signal, currently `BUY` or `SELL`. |
| `price` | Simulated execution price. |
| `quantity` | Simulated quantity. |
| `cash_after` | Simulated cash after the trade. |
| `position_after` | Simulated position after the trade. |
| `metadata` | Optional non-critical JSONB details. |
| `created_at` | Row creation timestamp. |

Primary key:

- `id`.

Foreign keys:

- `backtest_run_id` references `backtest_runs(id)`.

Uniqueness:

- Unique `backtest_run_id + sequence`.

Indexes:

- Composite index on `backtest_run_id, sequence` for ordered trade reads.
- Composite index on `backtest_run_id, candle_open_time` for overlaying trade markers on graphs.
- Optional index on `backtest_run_id, signal` if later reads filter BUY and SELL markers separately.

Notes:

- These trades are simulated backtest trades only. They are not exchange orders and must not include real order IDs, account IDs, signed-request data, or API-key material.

## `backtest_graph_points`

Owner role: Backtest/Persistence.

Purpose: Store dense graphable time-series output so later graphing can read equity, close price, position, cash, and trade-marker context without re-running a backtest.

Fields:

| Field | Meaning |
| --- | --- |
| `id` | Surrogate primary key. |
| `backtest_run_id` | Owning run. |
| `sequence` | Deterministic candle order within the run. |
| `candle_open_time` | Candle open timestamp for the point. |
| `close_price` | Candle close price used for mark-to-market equity. |
| `cash` | Simulated cash after applying any trade at this candle. |
| `position` | Simulated position after applying any trade at this candle. |
| `equity` | `cash + position * close_price` after applying any trade at this candle. |
| `trade_id` | Nullable foreign key to the simulated trade at this candle when a trade occurred. |
| `signal` | Nullable graph marker signal, currently `BUY` or `SELL` when a trade occurred; null for non-trade candles. |
| `metadata` | Optional non-critical JSONB graph metadata. |
| `created_at` | Row creation timestamp. |

Primary key:

- `id`.

Foreign keys:

- `backtest_run_id` references `backtest_runs(id)`.
- `trade_id` references `backtest_trades(id)` and is nullable.

Uniqueness:

- Unique `backtest_run_id + sequence`.
- Unique `backtest_run_id + candle_open_time` because stored candle inputs must be unique by open time within source/symbol/interval.

Indexes:

- Composite index on `backtest_run_id, candle_open_time` for graph reads ordered by time.
- Composite index on `backtest_run_id, sequence` for deterministic replay/checking.
- Optional partial index on `backtest_run_id, candle_open_time` where `signal IS NOT NULL` can be added later if trade marker reads become separate from full graph reads.

Graph read behavior:

- A later graph workflow should query `backtest_runs` by `run_key` or latest-run filters, join `backtest_results` for summary metrics, read `backtest_graph_points` ordered by `candle_open_time`, and optionally read `backtest_trades` ordered by `sequence` for richer marker details.
- The graph workflow should not call `RsiStrategy`, `BasicBacktester`, Binance endpoints, or any exchange account/order APIs.
- Empty backtest runs have a `backtest_results` row with zeroed summary values and no `backtest_graph_points` rows.

## Relationship Summary

- `strategy_configs 1 -> many backtest_runs`.
- `backtest_runs 1 -> 1 backtest_results`.
- `backtest_runs 1 -> many backtest_trades`.
- `backtest_runs 1 -> many backtest_graph_points`.
- `backtest_graph_points many -> 0/1 backtest_trades` through nullable `trade_id` for marker context.
- Candle rows are not owned by backtest output tables. A run references candle input identity by source, symbol, interval, requested time range, actual time range, and candle count rather than by copying candle row IDs into every graph point.

## Expected Task 022 Write Behavior

The next implementation task should keep writes atomic:

1. Resolve or insert the deterministic `strategy_configs` row.
2. Compute `run_key` from canonical run identity inputs.
3. Insert or idempotently replace the `backtest_runs` row.
4. Insert exactly one `backtest_results` row.
5. Insert ordered `backtest_trades` rows.
6. Materialize ordered `backtest_graph_points` rows from the run candles and returned trades.
7. Commit all output rows together, or roll back all output rows on failure.
8. Continue printing the deterministic stdout JSON payload unless the task explicitly changes CLI output.

Task 022 should decide whether saving is opt-in via a CLI flag, but database writes must not become hidden live-trading or exchange-order behavior.

## Expected Task 023 Read Behavior

The read model should support at least:

- latest completed runs by created time,
- completed runs by source, symbol, interval, and candle time range,
- one run's summary metrics,
- one run's ordered simulated trades,
- one run's ordered graph points with `candle_open_time`, `close_price`, `cash`, `position`, `equity`, and nullable marker fields.

The read model should not require PostgreSQL-specific logic in strategy classes, market-data providers, or graph rendering code beyond a repository/read-model boundary.

## Contract Test Expectations For Later Implementation

Ordinary tests for Task 022 and Task 023 should not require real PostgreSQL, Docker, Binance, or external network access. They should use fakes, mocks, or pure functions for canonicalization and materialization where possible.

Expected contract tests:

- Deterministic run-key generation is stable for canonical equivalent inputs and changes when strategy parameters, input identity, candle range, starting cash, or trade quantity changes.
- Strategy config parameter canonicalization is deterministic and includes RSI `window`, `buy_threshold`, and `sell_threshold`.
- Result persistence preserves the standard candle contract: candles provided to strategy/backtest still contain only `timestamp`, `open`, `high`, `low`, `close`, and `volume`.
- `BasicBacktester.run(candles, strategy)` public signature remains unchanged.
- Graph-point materialization emits one ordered point per input candle and does not call the strategy again.
- Graph-point materialization applies simulated trades at the matching candle timestamp and computes equity as `cash + position * close_price`.
- Empty runs save a summary and no graph points.
- Trade rows are ordered and uniquely identified by `backtest_run_id + sequence`.
- Read-model queries return graph points ordered by `candle_open_time` and trades ordered by `sequence`.

## Safety Test Expectations For Later Implementation

Expected safety tests:

- Persistence and graph read tests do not call Binance network endpoints.
- No exchange account API or order endpoint calls are introduced.
- No API keys, signed-request headers, real order IDs, `.env` files, or live-trading toggles are persisted or required.
- Paper/backtest trades remain explicitly simulated and are not treated as real execution records.
- CLI database-save behavior, if added, is distinct from live trading and does not place orders.
