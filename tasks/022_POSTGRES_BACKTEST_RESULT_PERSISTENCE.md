# Task 022: PostgreSQL Backtest Result Persistence

# Goal

Implement persistence for `quant-bitcoin-postgres-backtest` so each completed PostgreSQL-backed RSI backtest can save its run metadata, summary, simulated trades, and graph-ready time-series output to PostgreSQL.

# Source Requirement

User request on 2026-05-16:

- `quant-bitcoin-postgres-backtest` should backtest and insert the result into the database.
- The database should be designed so a later workflow can read it and draw graphs.
- Follow `AGENTS.md` and define the next tasks before implementation.

# Extracted Roles

- Owner role: Backtest Engine
- Supporting roles: Persistence, Strategy metadata, Market Data Provider input, Test Designer, Documentation
- Forbidden roles: Live Trading, Real Binance order execution, Exchange account APIs, Dashboard, FastAPI, Streamlit, Scheduler, Risk Management, Portfolio Optimization, Machine Learning

# Context

Task 020 made `quant-bitcoin-postgres-backtest` runnable. It currently reads candles from PostgreSQL, runs the existing RSI strategy with the existing basic backtester, and prints deterministic JSON. Task 021 must first define the accepted graph-ready persistence schema for backtest outputs.

This task should implement only the accepted write path and keep the existing command easy to run locally. PostgreSQL candle rows remain read-only inputs. Persisted backtest rows are simulated historical outputs only; they are not orders and must not trigger any exchange behavior.

# Scope

- Implement the accepted Task 021 schema in the project's existing PostgreSQL initialization path.
- Add persistence repository methods for creating a backtest run, storing summary results, storing simulated trades, and storing graph-ready time-series points.
- Wire `quant-bitcoin-postgres-backtest` to persist completed results to PostgreSQL.
- Preserve deterministic JSON stdout output for local inspection.
- Include the persisted run identifier in stdout JSON after a successful DB write.
- Ensure DB writes happen only for simulated backtest outputs generated by the local backtest engine.
- Add tests with mocked/fake persistence behavior so ordinary tests do not require real PostgreSQL, Docker, Binance, or external network access.
- Update README usage documentation for saved backtest runs.
- Update `STATUS.md` during implementation if phase, step, goal, active task, blocker, open question, or completion state changes.

# Out of Scope

- Graph rendering or graph UI.
- Read APIs for graph consumers beyond the minimal repository reads needed by tests, unless Task 021 explicitly requires them for verification.
- FastAPI, Streamlit, dashboard, scheduler, or daemon behavior.
- Live trading.
- Real Binance order execution.
- Exchange account APIs.
- API keys, signed requests, tracked `.env` files, or real account state.
- Strategy algorithm changes.
- Market data ingestion changes.
- Risk management, futures, leverage, portfolio optimization, or machine learning.
- Changing `BasicBacktester.run(candles, strategy)` unless Task 021 records a blocker and the project owner explicitly assigns a contract-change task.

# Acceptance Criteria

- `quant-bitcoin-postgres-backtest` persists completed backtest output to PostgreSQL using the accepted Task 021 schema.
- Persisted output includes run metadata, strategy parameters, summary metrics, simulated trades, and graph-ready time-series points.
- The command still prints deterministic JSON to stdout.
- The stdout JSON includes the persisted backtest run id after a successful DB write.
- Database writes are transactional or otherwise all-or-nothing for a run.
- Tests cover CLI wiring to persistence with mocked/fake repository behavior.
- Tests cover deterministic trade sequence persistence.
- Tests cover graph point persistence shape and ordering.
- Tests cover empty-candle behavior if the runner persists empty-run summaries.
- Ordinary tests do not require real PostgreSQL, Docker, Binance, or external network access.
- No live-trading, signed request, API-key, `.env`, exchange account API, or order endpoint behavior is added.
